## 深层 Root Cause 挖掘（Deep Root Cause Drill）

> 本文档在阶段 1→6 初始分析完成后**立即执行**。  
> 唯一目标：找到**最深层的、不可再分解的代码缺陷**。  
> 不做利润计算、不做资金流表格、不做 LP 损失估算——这些是表层工作，不在本阶段范围内。

---

## 核心原则

**初始分析（阶段 1→6）找到的 root cause 几乎总是"表层 root cause"。** 它能解释"攻击者做了什么"，但往往没有回答"为什么攻击者能做到这一点"。

举例：
- 表层："`TokenGateway` 没有校验 `request.from`，所以攻击者能发治理消息"  
  → 深层追问：**攻击者的消息是怎么通过 proof verification 的？proof verifier 的代码是否正确？**
- 表层："`deposit` 在 flash 窗口内读到了偏小的 `grossLiquidity`"  
  → 深层追问：**为什么 `flash` 没有锁？`flash` 的设计者是否考虑过这个场景？被注释掉的 invariant check 如果存在能否阻止？**

**你必须沿着因果链一直追问，直到碰到"这行代码就是 bug"为止。**

---

## 执行流程（严格按顺序）

### 第 1 步：画出信任边界链

从攻击的入口点到最终的状态写入，列出路径上的每一个**信任边界跨越**：

```
入口 → [边界1: 谁校验了什么？] → 中间层 → [边界2: 谁校验了什么？] → ... → 最终写入
```

对每个边界，写出：
- **校验函数名 + 所在文件**
- **校验逻辑的一句话描述**（如"检查 msg.sender == owner"、"验证 Merkle proof"、"检查 request.source == hyperbridge"）

### 第 2 步：逐个边界打开源码审查

**从最外层（离入口最近的）开始，逐个打开校验函数的完整源码。**

对每个校验函数，你必须：
1. **读完整源码**（不是只看函数签名，要读函数体的每一行）
2. 回答以下问题：
   - 这个校验是否能被绕过？如何绕过？
   - 输入参数是否有未校验的边界条件？（如 `index >= length`、`length == 0`、`proof.length == 0`、空数组、零地址）
   - 如果涉及密码学/哈希/Merkle，proof 计算是否正确地把**被验证的数据**绑定到了**验证结果**？（即：修改被验证的数据是否一定会导致验证失败？）
   - 是否存在 replay 向量？（同一个 proof/signature 能否用于不同的 request/message？）

3. **不允许跳过任何校验函数。** 即使上一层的分析"已经能闭环解释攻击"，你仍然必须检查下一层。原因：
   - 上层的修复建议可能是错的（如果底层也有 bug，修上层也没用）
   - 真正的 root cause 可能在更底层
   - 安全团队需要知道**所有**被利用的代码缺陷，不只是最表面那个

### 第 3 步：对每个边界给出结论

对每个信任边界，给出以下之一：
- **SECURE**：校验逻辑正确，攻击者无法绕过。给出理由。
- **VULNERABLE**：校验逻辑有缺陷。给出：
  - 具体哪行代码有问题
  - 缺陷的精确描述（缺少什么检查？数学/逻辑错误在哪？）
  - 攻击者如何利用这个缺陷
- **[OPEN]**：源码不可用或无法确定。给出需要什么才能确定。

### 第 4 步：确定最深层 root cause

在所有标记为 VULNERABLE 的边界中，找到**最底层的那个**（即：如果修复了这个 bug，上层的所有 bug 都不再可被利用）。

这就是最深层 root cause。如果有多个独立的 VULNERABLE 边界（互不依赖），全部列出。

### 第 5 步：修正初始分析的结论

如果最深层 root cause 与阶段 3→6 的结论不同，**必须更新阶段 6 的 root cause 描述**，补充为：
- **表层 root cause**：（阶段 6 原始结论）
- **最深层 root cause**：（本步骤发现的）
- **关系**：最深层 bug 使得表层 bug 可被利用（或：两者是独立的 bug，都需要修复）

## 常见的"假停止点"（你必须避免的陷阱）

1. **"应用层鉴权缺失"不是终点。** 如果攻击者提交了一条"看起来合法"的消息/proof，你必须验证 verification 层的代码是否真的正确。
2. **"proof 通过了所以是合法的"不是理由。** proof verification 代码本身可能有 bug（输入校验缺失、边界条件错误、绑定断裂）。
3. **"候选 A 已能闭环"不是跳过候选 B 的理由。** 安全审计要求检查所有攻击面，而不是找到第一个解释就停止。
4. **"源码太复杂/太长"不是不读的理由。** 读完整源码。如果函数超过 100 行，分段读。
5. **"[OPEN] 需要更多数据"** 只有在源码确实不可用时才允许使用。如果源码在 `contract_sources/` 里有，你必须读它。
